# Header banner: pre-baked half-block image, no runtime image dependency

The header renders a fixed decorative image in place of the `CLAUDE_SPARK` art by emitting **Unicode half-block cells** (`▀`, top pixel = truecolor foreground, bottom pixel = truecolor background — two vertical pixels per character cell). A future reader will ask three "why not the obvious thing" questions; this records the answers.

**Why half-blocks, not a native graphics protocol (iTerm/Kitty/Sixel).** The header's render contract is `render(width: number): string[]` — synchronous, line-oriented, re-run on every resize, with each returned line passed through `truncateToWidth`. Native protocols draw outside the character grid via raw escape sequences that have no honest column width, so pi's per-line truncation and resize redraws would shred them. Half-blocks live entirely inside the character grid, so they survive truncation and resize like any other styled line. This also confines us to truecolor terminals (see Consequences).

**Why pre-baked, not decoded at runtime.** The image is a single fixed asset (`assets/pokemon_1.jpg`, 1162×941). Decoding it at runtime would force a heavy image dependency (`jimp`, or `sharp` with a native binary) into every install plus async decode work in `session_start`. Instead the asset is converted **once**, by a regen script, into a committed **`.ts` module** — `export const BANNER: string[]` of 12 finished-ANSI lines (24px tall ÷ 2 px/cell = 12 rows; ~30 cols at the source aspect ratio, half-block sub-pixels being square so there is no distortion). We chose a `.ts` module over a `.txt` asset specifically to dodge runtime path resolution and `fs`: it is imported like `lib.ts` and survives any git/npm distribution as plain module code, whereas a packaged `.txt` is exactly the kind of asset that resolves locally and breaks when installed. The generated file lands in `packages/header/`, so it is inside `npm run check` (type-checked + linted); the generator therefore emits `string[]`-typed output using `\u001b` **escape sequences** (via `JSON.stringify`), never raw ESC bytes. At runtime the header just returns those 12 strings. The decoder (`sharp`, chosen over `jimp` for Lanczos quality at the ~39× downscale) lives only as a **devDependency** used by the regen script; the shipped extension has no image dependency. We accepted that "finished ANSI" makes the artifact opaque in diffs and leaves nothing to unit-test — deliberately, because the image *content* is not worth testing.

**The regen script is a `.mjs`, run manually, outside `check`.** It lives at `scripts/bake-header-banner.mjs`, invoked via a documented `npm run bake:header` (`node scripts/bake-header-banner.mjs`). Plain JS, not `.ts`: it is dev-only and never imported by the extension, so keeping it out of the `**/*.ts` lint glob and `tsconfig.include` avoids wiring `scripts/**` into the typed project and picking a `.ts` runner (no `tsx`/`ts-node` is installed). There is **no drift guard** — `check` does not verify the artifact is fresh. Staleness (image swapped, artifact not re-baked) is prevented by manual discipline and this documented command, not by pulling native `sharp` into every `check`/CI run.

**Why chop, not reflow; why no spark fallback.** The banner is pre-rendered at its natural ~30-column width. On a terminal narrower than that, the banner rows are clipped on the right edge — we do **not** re-scale or reflow, and we do **not** fall back to the spark. The image always renders; at, say, 20 columns two-thirds of it is simply gone. Clipping uses `truncateToWidth(line, width, "")` with an **empty ellipsis**: the default `"..."` would splatter literal dots into the image, so banner rows and text rows (subtitle/sections) can no longer share one truncation call — `render()` splits them. `truncateToWidth` appends `\x1b[0m` at the cut, so a clipped half-block row does **not** bleed its background color across the rest of the terminal line. The `CLAUDE_SPARK` art is removed entirely. The `claude · pi vVERSION` subtitle and the `[Context]/[Skills]/[Extensions]` sections still render below the banner, unchanged.

## Consequences

The header now costs ~20 lines before any conversation (12 banner rows + subtitle + section block), roughly double the built-in — accepted as a deliberate aesthetic choice. The banner assumes 24-bit color; on a 256-color-only terminal the truecolor escapes may render approximately or wrong, and there is no degraded palette path. Changing the image (or its size) means running `npm run bake:header` and re-committing `banner.ts` — there is no runtime path to swap it, and no automated check that it is current, by design. If the image ever needs to be user-configurable or theme-tinted, this decision must be reopened: that reintroduces a runtime decoder and an RGB-grid artifact with a testable `grid → ANSI` seam (the alternative we rejected here).
