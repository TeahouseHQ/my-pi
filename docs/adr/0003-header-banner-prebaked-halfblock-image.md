# Header banner: pre-baked half-block image, no runtime image dependency

The header renders a fixed decorative image in place of the `CLAUDE_SPARK` art by emitting **Unicode half-block cells** (`▀`, top pixel = truecolor foreground, bottom pixel = truecolor background — two vertical pixels per character cell). Transparent source pixels are left **unpainted** so the terminal background shows through rather than being flattened to a solid colour (see "Why leave transparency unpainted" below). A future reader will ask several "why not the obvious thing" questions; this records the answers.

**Why half-blocks, not a native graphics protocol (iTerm/Kitty/Sixel).** The header's render contract is `render(width: number): string[]` — synchronous, line-oriented, re-run on every resize, with each returned line passed through `truncateToWidth`. Native protocols draw outside the character grid via raw escape sequences that have no honest column width, so pi's per-line truncation and resize redraws would shred them. Half-blocks live entirely inside the character grid, so they survive truncation and resize like any other styled line. This also confines us to truecolor terminals (see Consequences).

**Why pre-baked, not decoded at runtime.** The image is a single fixed asset (`assets/pokemon.jpg`, a 1197×850 photo of **pixel art drawn on a grid**). Decoding it at runtime would force a heavy image dependency (`jimp`, or `sharp` with a native binary) into every install plus async decode work in `session_start`. Instead the asset is converted **once**, by a regen script, into a committed **`.ts` module** — `export const BANNER: string[]` of finished-ANSI lines (one half-block line per two bitmap rows). We chose a `.ts` module over a `.txt` asset specifically to dodge runtime path resolution and `fs`: it is imported like `lib.ts` and survives any git/npm distribution as plain module code, whereas a packaged `.txt` is exactly the kind of asset that resolves locally and breaks when installed. The generated file lands in `packages/header/`, so it is inside `npm run check` (type-checked + linted); the generator therefore emits `string[]`-typed output using `\u001b` **escape sequences** (via `JSON.stringify`), never raw ESC bytes. At runtime the header just returns those strings. The decoder (`sharp`, chosen over `jimp` for its raw-pixel access) lives only as a **devDependency** used by the regen script; the shipped extension has no image dependency. We accepted that "finished ANSI" makes the artifact opaque in diffs and leaves nothing to unit-test — deliberately, because the image *content* is not worth testing.

**Why reconstruct the grid, not downscale the photo.** The asset is not a native-resolution sprite but a *photo of pixel art on a grid* — each grid cell is one intended art pixel, drawn much larger with faint grey gridlines between cells. Naively downscaling it (`resize` to ~24px tall) would blur across gridlines and cell boundaries, muddying the crisp palette. Instead the baker recovers the true bitmap: it detects the grid **period** by autocorrelating the projection of gridline-coloured pixels and the **phase** by the offset that best aligns to them, then samples the **median** colour of an inner patch at each cell centre (median rejects the gridlines and JPEG ringing at cell edges). This yields the exact NxM art bitmap (currently 24×17), one bitmap pixel per cell, which then pairs into half-block rows. The grid geometry is auto-detected rather than hard-coded so re-shooting or swapping the art doesn't require editing the script.

**The regen script is a `.mjs`, run manually, outside `check`.** It lives at `scripts/bake-header-banner.mjs`, invoked via a documented `npm run bake:header` (`node scripts/bake-header-banner.mjs`). Plain JS, not `.ts`: it is dev-only and never imported by the extension, so keeping it out of the `**/*.ts` lint glob and `tsconfig.include` avoids wiring `scripts/**` into the typed project and picking a `.ts` runner (no `tsx`/`ts-node` is installed). There is **no drift guard** — `check` does not verify the artifact is fresh. Staleness (image swapped, artifact not re-baked) is prevented by manual discipline and this documented command, not by pulling native `sharp` into every `check`/CI run.

**Why leave transparency unpainted, not flatten to a background colour.** The pixel art has no alpha channel — its "background" is the white grid. A half-block cell can only paint solid colours, so painting those white cells would box the sprite in a white rectangle that only matches one terminal theme. Instead the baker derives transparency geometrically: it flood-fills white cells inward from the image border and treats that border-connected white region as background (transparent), while white cells *enclosed* by art (eyes, highlights) stay opaque. Transparent bitmap pixels are then left unpainted — a cell with both pixels transparent bakes to a bare space over the default background (`\u001b[49m`), and a cell with one transparent pixel uses `▀` or `▄` with only the opaque half coloured and the other half left as the default background. The image therefore floats on whatever the terminal background is, with no baked-in rectangle — at the cost of a hard classification (a cell is fully art or fully background, no partial-coverage edge blending against an unknown background, which we can't compute at bake time).

**Why chop, not reflow; why no spark fallback.** The banner is pre-rendered at its natural ~24-column width (one column per art pixel). On a terminal narrower than that, the banner rows are clipped on the right edge — we do **not** re-scale or reflow, and we do **not** fall back to the spark. The image always renders; at, say, 20 columns two-thirds of it is simply gone. Clipping uses `truncateToWidth(line, width, "")` with an **empty ellipsis**: the default `"..."` would splatter literal dots into the image, so banner rows and text rows (subtitle/sections) can no longer share one truncation call — `render()` splits them. `truncateToWidth` appends `\x1b[0m` at the cut, so a clipped half-block row does **not** bleed its background color across the rest of the terminal line. The `CLAUDE_SPARK` art is removed entirely. The `claude · pi vVERSION` subtitle and the `[Context]/[Skills]/[Extensions]` sections still render below the banner, unchanged.

## Consequences

The header now costs ~17 lines before any conversation (9 banner rows + subtitle + section block), well above the built-in — accepted as a deliberate aesthetic choice; the exact banner height follows the source art (currently a 24×17 bitmap → 9 half-block rows). The banner assumes 24-bit color; on a 256-color-only terminal the truecolor escapes may render approximately or wrong, and there is no degraded palette path. Changing the image (or its size) means running `npm run bake:header` and re-committing `banner.ts` — there is no runtime path to swap it, and no automated check that it is current, by design. If the image ever needs to be user-configurable or theme-tinted, this decision must be reopened: that reintroduces a runtime decoder and an RGB-grid artifact with a testable `grid → ANSI` seam (the alternative we rejected here).
